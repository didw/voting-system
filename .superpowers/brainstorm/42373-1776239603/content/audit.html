시스템 전체 결함 분석

전체 코드베이스를 분석하여 발견된 문제점 — 심각도 순 정렬

🔴 Critical — 기능 불가

① 행운권 추첨: 애니메이션 숫자 ≠ 실제 당첨 숫자

LotteryWheel이 시각적으로 보여주는 숫자와 서버가 최종 선택한 숫자가 다릅니다. 화면에 "47"이 스피닝으로 멈춰도, 실제 API 응답에서 "183"이 나올 수 있습니다. 극적 연출 전체가 무의미해집니다.

② Ranking: 진행 중인 세션도 순위에 노출

GET /api/teams가 status 무관하게 전체 반환 → 현재 투표 중인 팀(점수 0)도 순위표에 포함됩니다. status = 'finished' 필터 없음.

③ 투표 중 WebSocket 끊김 시 투표 수 업데이트 중단

투표 진행 중 WS가 끊기면 실시간 투표 카운트가 멈춥니다. 폴백 폴링이 없어, 투표가 진행되고 있어도 화면은 마지막 카운트에서 고정됩니다.

🟡 High — UX 심각 저하

④ Devices 상태 임계값이 실제 사용 패턴과 불일치

현재: 2분 이내 = 연결됨. 그런데 기기는 10~20분에 한 번 투표 신호를 보냅니다. 결과: 정상 작동 중인 기기가 항상 "끊김"으로 표시됩니다.

⑤ 투표/기기 페이지에 WS 연결 상태 표시 없음

WebSocket 연결이 끊겨도 UI에 아무런 경고가 없습니다. 관리자가 실시간 연결 상태를 알 수 없어 장애 발생 시 즉각 대응 불가.

⑥ 투표 시작 버튼 연속 클릭 시 중복 세션 생성

startVoting() 에 loading 상태 처리 없음 → 빠르게 여러 번 클릭하면 동일 팀 이름으로 여러 세션이 DB에 삽입됩니다.

⑦ 페이지 이탈 시 활성 세션이 'voting' 상태로 무한 고착

투표 중 다른 탭으로 이동하거나 창을 닫으면 해당 세션은 'voting' 상태로 DB에 남습니다. 다음 팀 투표 시 두 개의 active session이 존재할 수 있습니다.

⑧ 전체 리셋 시 기기 목록까지 삭제

DELETE /api/teams가 sessions/votes/results와 함께 devices 테이블도 삭제합니다. 기기는 세션 데이터가 아님에도 함께 지워집니다.

🔵 Medium — 안정성

⑨ Gateway WebSocket — ping/pong heartbeat 없음

네트워크 장애 후 클라이언트가 강제 종료되면 gateway의 wss.clients에 죽은 연결이 축적됩니다. 재연결도 고정 2초 간격으로 지수 백오프 미적용.

⑩ API 오류 시 UI에 아무런 피드백 없음

fetch 실패/서버 오류를 catch만 하고 사용자에게 알리지 않습니다. 상용 수준에서는 toast/snackbar 알림 필수.

⑪ Judge 페이지 saveAll()이 순차 실행

팀이 많을 경우 for 루프로 API를 순차 호출합니다. Promise.all로 병렬 처리 필요.
